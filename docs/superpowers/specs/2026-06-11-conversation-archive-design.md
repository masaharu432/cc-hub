# 会話アーカイブ(非表示)機能 設計

日付: 2026-06-11
ステータス: 承認待ち

## 背景 / 目的

一覧(overview)は「稼働中の tmux セッション」と「停止中の再開可能な会話」
(`~/.claude/projects/**/<sid>.jsonl` 由来)の2種類で構成される。稼働中は
✕(kill)で終了できるが、停止中の会話は消す手段がなく、使い捨てセッションが
一覧に溜まり続ける。これを**非破壊で一覧から隠せる**ようにする。

## 決定事項(ユーザー確認済み)

1. **アーカイブ(非表示)のみ** — jsonl は削除しない。CLI の resume 履歴は無傷。
   古いログの実削除は Claude 本体の自動掃除(`cleanupPeriodDays`、既定30日)に任せる。
2. **アーカイブタブで復元可能** — 既存の拡張可能タブUIに「アーカイブ」タブを追加。
3. **kill とは独立** — ✕ は従来どおり「停止して再開可能欄に移す」だけ。
4. **操作はスワイプ + PC幅ではボタン併設** — 行のフルスワイプで即アーカイブ(Gmail風)。
   スマホ幅ではボタンを置かないが、画面に余裕がある PC 幅(Bootstrap md, ≥768px)
   では行にアーカイブボタンも表示する。確認ダイアログなし。undo トーストも置かない
   (復元はアーカイブタブに一本化 — 2026-06-11 ユーザー判断で undo 案を撤回)。
   実行後は通常の成功トーストで「アーカイブタブから戻せる」ことだけ伝える。

## データモデル

- `config.json` の隣に **`archive.json`** を新設:
  `{"archived": ["<session-id>", ...]}`
- 位置づけ: セッション状態ではなく**表示上のビュー状態**。
  「tmux が唯一の真実 / 独自セッションDBなし」の原則には抵触しない。
- 書き込みは tmp ファイル + `os.replace` のアトミック書き(`ensure_trusted` と同パターン)。
  ThreadingHTTPServer 下で並行アクセスされるため `threading.Lock` で read-modify-write を保護。
- **自己掃除**: 読み込み時に、対応する jsonl が存在しない id
  (Claude 本体の自動掃除で消滅したもの)をリストから除去して書き戻す。
  ファイルは無限に育たない。
- 壊れた/読めない archive.json は空扱いにしてログを出す(起動不能にしない)。
- **SQLite を採用しない理由**: 書き込み主体は本サーバプロセス1つだけで
  (同一ポートの bind により二重起動は不可能)、スレッド間競合は in-process の
  Lock で完結する。SQLite が要る「複数プロセスの同時書き込み」は存在せず、
  データはUUIDの小リスト・低頻度書き込みなので、スキーマ/コネクション管理の
  固定費に見合わない。複数プロセス化、またはアーカイブにメタデータや検索が
  必要になった時点で再検討する。

## サーバ API

| エンドポイント | 動作 |
|---|---|
| `POST /api/archive` `{id}` | id を追加。UUID 検証、冪等。**稼働中の id は 400 で拒否**(kill 後に突然消える混乱を防ぐ) |
| `POST /api/unarchive` `{id}` | id を削除。冪等 |
| `GET /api/archived` | アーカイブ済み会話のメタデータ一覧(既存 `_read_conversation` を再利用、新しい順) |

既存エンドポイントの変更:

- `build_overview()` — 再開可能リストからアーカイブ済み id を除外する。
  稼働中セッションは常に表示(アーカイブ対象外)。
- `resume_session()` — アーカイブ済み会話を再開したら**自動 unarchive**
  (再開=「戻したい」という意思。再 kill 後にまた勝手に消えるのを防ぐ)。
- `search_conversations()` の HTTP 層 — 結果に `archived: true/false` を付与。
  アーカイブ済みも検索には出す(隠れたまま見つからなくなるのを防ぐ)。

## フロントエンド

### スワイプ・トゥ・アーカイブ(停止中の行のみ)

- **ジェスチャ**: Pointer Events(`pointerdown/move/up` + `setPointerCapture`)で
  タッチとマウスドラッグの両方に対応。
- **縦スクロールとの共存**: 行に `touch-action: pan-y`。横移動が約10pxを超え、
  かつ |dx| > |dy| のときだけスワイプ開始(方向ロック)。それ以外はスクロールに譲る。
- **追従表示**: スワイプ中は行を `translateX` で追従させ、下からアーカイブ色の
  背景 + `bi-archive` アイコンが見える。
- **確定条件**: 指を離した時点で |dx| ≥ 行幅の50%、または速いフリック。
  未達ならスプリングバック(transition で 0 に戻す)。
- **確定時**: 行をスライドアウト→ `POST /api/archive` →「アーカイブしました
  （アーカイブタブから戻せます）」の成功トースト(自動消滅)→一覧を再取得。
- 稼働中の行はスワイプ対象外(サーバ側でも 400 で拒否される)。

### PC幅でのアーカイブボタン

- 停止中の行に `bi-archive` ボタンを追加し、CSS(`d-none d-md-inline-flex` 相当)で
  **≥768px のときだけ表示**。クリックでスワイプ確定と同じ処理
  (`POST /api/archive` + undo トースト)を呼ぶ。
- スマホ幅ではスワイプのみ(ボタン非表示で行を圧迫しない)。
- マウスドラッグでのスワイプも全幅で引き続き有効(Pointer Events のため自然に対応)。

### アーカイブタブ

- 既存タブUI(検索タブと同じ仕組み)に「アーカイブ」タブを追加。
- フラットに新しい順で表示。各行に: タイトル / プロジェクト名 / 最終入力の抜粋。
- 復元手段は2系統: **スワイプで復元**(同ジェスチャ)+ 明示的な「戻す」ボタン
  (スワイプは発見性が低いため両方置く)。
- 行から直接「再開」も可能(再開すれば自動 unarchive)。
- 空のときは「アーカイブはありません」を表示。

### 検索タブ

- アーカイブ済みのヒットには「アーカイブ済」バッジを表示。そこから再開すれば
  自動 unarchive される。

## エラー処理

- `archive.json` 読み込み失敗 → 空扱い + サーバログ(`_log`)。
- `POST /api/archive` で UUID 不正 / 稼働中 id → 400 + メッセージ。
- unarchive は常に冪等成功(存在しない id でも 200)。

## テスト計画

1. サーバ再起動(必須 — 長寿命プロセスのため)。
2. curl で API 検証: archive → overview から消える / archived 一覧に出る /
   unarchive → overview に戻る / 稼働中 id の archive が 400 /
   resume で自動 unarchive。
3. ブラウザ実機(スマホ)で: スワイプの追従・スプリングバック・縦スクロール非干渉、
   トーストの undo、アーカイブタブの復元、検索バッジ。

## 非スコープ

- jsonl の実削除(完全削除機能)。必要になったら別途設計。
- 稼働中セッションのアーカイブ(kill+archive の一括操作)。
- アーカイブの自動期限切れ(Claude 本体の掃除で間接的に実現される)。
